3:T1d37,<h1>High-Level Kubernetes Architecture</h1>
<h1></h1>
<p><a href="Kubernetes-Physical-Architecture.png"><img src="Kubernetes-Physical-Architecture.png" alt=""></a></p>
<h2>What Kubernetes is?</h2>
<p>Open-source container orchestration tool originally developed by Google. It helps managing applications that made up hundreds of thousands of containers and help to manage in different environments (physical, VM, cloud or hybrid).  </p>
<h2>What problem does it solve?</h2>
<p>The application development trend is moving from monolith to micro-services. Increased usage of containers using scripts and self-made tools in different environments is very complex. Demand for a proper way of managing those hundreds of containers raised the need to have container orchestration technologies. Kubernetes provides features like  </p>
<ul>
<li>High Availability</li>
<li>Scalability</li>
<li>Disaster recovery</li>
</ul>
<h3>Kubernetes cluster</h3>
<ul>
<li>Consist of multiple nodes. </li>
<li>Set of worker machine nodes. Cluster shares the load. </li>
<li>If one node goes down other nodes keep the application available.</li>
</ul>
<h3>Nodes</h3>
<ul>
<li><p>A server or virtual machine that can host k8s pod.  </p>
</li>
<li><p>Each node can have multiple pods on it</p>
</li>
</ul>
<h3>Worker Node</h3>
<ul>
<li><p>A machine responsible to run containers. Have below Kubernetes components. Actually do the work</p>
</li>
<li><p>Pod  </p>
</li>
<li><p>Abstraction over containers</p>
</li>
<li><p>Host container runtime environment</p>
</li>
<li><p>Can have multiple container per pod but usually 1 container per pod</p>
</li>
<li><p>Runs container and some side service that is required to host the pod</p>
</li>
<li><p>Each pod gets its own IP address, pod communicate with each other using IP address.</p>
</li>
<li><p>If pod dies and re-created a new IP address is assigned. On such events it is very hard to communicate with pods. To come over of this situation service is used.</p>
</li>
<li><p>Service</p>
</li>
<li><p>Service have permanent IP address.</p>
</li>
<li><p>Every container pod will have its own service</p>
</li>
<li><p>Life-cycle of service and pod are not connected, so even if pod dies service and its IP address stay. This eases pod communication.</p>
</li>
<li><p>External Services open communication with pod to external world. Internal service allows pod to pod communication and helps to keep pods private with-in K8s cluster if not required to expose externally. Ingress</p>
</li>
<li><p>Internal DNS for K8s cluster. Request first goes to ingress and it does forwarding to service IP address  </p>
</li>
<li><p>Kubelet</p>
</li>
<li><p>Ensure the number of replica sets</p>
</li>
<li><p>Container runtime</p>
</li>
<li><p>To execute container images. The popular runtime is docker. Kubernetes also supports Rkt-io &amp; CRI-O, containerd</p>
</li>
<li><p>Kube-Proxy</p>
</li>
</ul>
<h4>Master Node</h4>
<ul>
<li><p>Machine or set of machines responsible to run/execute below Kubernetes components. The master node is responsible for managing the cluster. If any node goes down the master node replaces it with another instance with the help of the controller.  </p>
</li>
<li><p>Manage worker node</p>
</li>
<li><p>Core Components  </p>
</li>
<li><p>ETCD</p>
</li>
<li><p>Key-value store. Externalize configuration.</p>
</li>
<li><p>Controller</p>
</li>
<li><p>The brain behind Kubernetes. Keep track of replica sets decides how to maintain the desired state.</p>
</li>
<li><p>Scheduler</p>
</li>
<li><p>Job Scheduler</p>
</li>
<li><p>Cron Scheduler</p>
</li>
<li><p>API Server</p>
</li>
<li><p>API Server is the gateway (frontend) to the Kubernetes cluster.</p>
</li>
<li><p>Exposes the REST interface to interact with Kubernetes components.</p>
</li>
<li><p>Validates and configure data for Kubernetes objects.</p>
</li>
<li><p>Command-line - kube-api [flag]</p>
</li>
<li><p>API Server components</p>
</li>
<li><p>API management - Expose REST API</p>
</li>
<li><p>API Path</p>
</li>
<li><p>API Transaction</p>
</li>
<li><p>API Discovery</p>
</li>
<li><p>Request processing - functionality that processes individual API request</p>
</li>
<li><p>Type of request [get | list | post | delete]</p>
</li>
<li><p>Life of request</p>
</li>
<li><p>AuthN</p>
</li>
<li><p>AuthZ / RBAC</p>
</li>
<li><p>Admission Control</p>
</li>
<li><p>Validate</p>
</li>
<li><p>Watch Operations</p>
</li>
<li><p>Encodings</p>
</li>
<li><p>Responses</p>
</li>
<li><p>Internal control loops - Perform necessary background operations for API Server.</p>
</li>
<li><p>Common Resource Definition(CRD) Control Loop</p>
</li>
</ul>

<h2>Logical Architecture</h2>
<h3>Deployment</h3>
<ul>
<li><p>Replica Set</p>
</li>
<li><p>Pod</p>
</li>
<li><p>Container Runtime</p>
</li>
<li><p>Image</p>
</li>
<li><p>Fluentd - realtime distributed log collection service</p>
</li>
</ul>
<h3>Service</h3>
<ul>
<li><p>Single access point for a group of pods</p>
</li>
<li><p>Provide consistent IP and port</p>
</li>
<li><p>Type of Service</p>
</li>
<li><p>NodePort</p>
</li>
<li><p>LocadBalancer</p>
</li>
</ul>
<h3>Volume</h3>
<ul>
<li><p>K8s doesn&#39;t manage data persistance, must be managed externally like backup or replication  </p>
</li>
<li><p>A dictionary that can hold data</p>
</li>
<li><p>Component of the pod and not independent of it</p>
</li>
<li><p>Create in the pod specification</p>
</li>
<li><p>Can&#39;t be deleted its own</p>
</li>
<li><p>Accessible to all container on a pod</p>
</li>
<li><p>Pod dies --&gt; volume dies</p>
</li>
<li><p>Advance than docker volume</p>
</li>
</ul>
